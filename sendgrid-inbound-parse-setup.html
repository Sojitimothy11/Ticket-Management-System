SendGrid Inbound Parse — Setup Guide

How to wire up the support inbox so incoming emails are converted into tickets by the POST /api/email/inbound/:secret route already implemented in this project.

1. Pick a dedicated inbound subdomain

SendGrid wants a hostname used only for inbound parsing — don't reuse a domain you send outbound mail from. For example, if your domain is yourdomain.com, use support.yourdomain.com.

2. Add the MX record

In your DNS provider, add:

Type:     MX
Host:     support.yourdomain.com
Value:    mx.sendgrid.net
Priority: 10

No SPF/DKIM/domain-authentication is needed for receiving mail — that's only required for outbound sending. Wait for DNS propagation, then verify:

nslookup -type=MX support.yourdomain.com

Confirm it resolves to mx.sendgrid.net.

3. Configure Inbound Parse in SendGrid

In the SendGrid dashboard: Settings → Inbound Parse → Add Host & URL

  • Receiving domain/hostname: support.yourdomain.com
  • Destination URL: https://<your-deployed-backend>/api/email/inbound/<INBOUND_EMAIL_SECRET>
  • POST the raw, full MIME message: leave off — the route expects parsed fields (text, html, subject, etc.), not raw MIME
  • Check incoming emails for spam: optional, your call
Note: INBOUND_EMAIL_SECRET is already generated in server/.env. Use that real value when building the Destination URL — never put the real secret in .env.example.

4. Set SUPPORT_EMAIL_ADDRESS

In server/.env, set it to match the address you'll actually receive mail at:

SUPPORT_EMAIL_ADDRESS="support@support.yourdomain.com"

This is the safety check in email.ts that ignores webhook calls not addressed to your support inbox.

5. Local development: expose your server publicly

SendGrid can't reach localhost. Use a tunnel while testing:

bun src/index.ts          # start your server on :3000
ngrok http 3000            # in a separate terminal

Take the https://xxxx.ngrok-free.app URL ngrok gives you and temporarily set it as the Destination URL in SendGrid's Inbound Parse settings:

https://xxxx.ngrok-free.app/api/email/inbound/<INBOUND_EMAIL_SECRET>
The free ngrok URL changes every restart, so you'll need to update the SendGrid setting each session. Point it at your real deployed URL once in production.

6. End-to-end test

Send a real email to support@support.yourdomain.com, then check:

  • Your server logs for the incoming POST
  • The Ticket and Message tables for the new row (e.g. via bunx prisma studio)

7. Production

When you deploy, update the Inbound Parse Destination URL to your real backend domain — the MX record, secret, and route logic all stay the same.
